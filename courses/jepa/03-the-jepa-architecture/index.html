That is the whole machine. One input gets split into a visible part and a hidden part, each half goes through its own encoder, a predictor guesses what the hidden half’s representation should be, and a loss measures how far the guess landed from the truth. Three boxes, one number out. The ╳ on the right branch is the one piece that looks strange, and we will earn it before the end.

This is the spine of the course. M1 made the bet — predict the representation of the hidden part, not its pixels — and M2 built the two tools the bet needs: an encoder, which turns an input into a representation (a vector), and distance, the number that says how far apart two representations sit. The diagram above is what you get when you wire those two tools into a learning system. Nothing here is new machinery; it is M2’s pieces, arranged.

The three boxes

The architecture has exactly three trainable boxes. Each is an encoder or a predictor — a function that takes one input and returns one output — and each has a single job.

The context encoder, fcf_cfc​. It reads the context: the visible part of the input, the part the system is allowed to see. Out comes a representation of that visible part — the context representation. This is the same kind of object M2 built: a fixed-length list of numbers.

The target encoder, ftf_tft​. It reads the target: the hidden part, the part the system must predict. Out comes the target representation, which we write sss. This is the answer key — the vector the system is trying to match. Crucially, ftf_tft​ gets to look at the hidden part directly. The rest of the system does not; it only ever sees the context.

The predictor, ggg. It reads the context representation that fcf_cfc​ produced and returns its best guess at what the target representation should be: the predicted representation, written g^\hat{g}g^​. The predictor is the one box doing the actual predicting — bridging from “here is what I can see” to “here is what I think is hidden.”

So the flow has a definite shape. The context goes fcf_cfc​ then ggg and lands as g^\hat{g}g^​, the guess. The target goes ftf_tft​ and lands as sss, the truth. Two paths, two vectors, and then a single comparison.

One thing the predictor does not produce: a picture. g^\hat{g}g^​ is a vector — a list of numbers, exactly the type M2 defined — and the system never turns it back into pixels. There is no decoder anywhere in this diagram. The guess and the answer are both points in representation space, and the only thing that ever happens to them is that we measure the distance between them.

The loss

The two paths meet at the loss. We have a guess, g^\hat{g}g^​, and an answer, sss, both vectors of the same length. We already know how to say how far apart two vectors are — that was M2’s whole job. Apply it.

Definition (JEPA loss)

The loss LLL is the squared distance between the predicted representation g^\hat{g}g^​ and the target representation sss:

L=∑i(g^i−si)2.L = \sum_i (\hat{g}_i - s_i)^2.L=i∑​(g^​i​−si​)2.

Here g^i\hat{g}_ig^​i​ is the iii-th number in the predicted representation, sis_isi​ is the iii-th number in the target representation, and the sum runs over every position in the vector.

Read it back against the picture. g^i−si\hat{g}_i - s_ig^​i​−si​ is, position by position, how far the guess sits from the answer; squaring makes every gap positive and punishes a large miss harder than a small one; summing collects all the positions into one number. That number is the squared distance from M2 — the same formula, now scoring a prediction. A small LLL means the guess landed close to the answer; a large LLL means it missed. Training pushes LLL down.

In code, one forward pass and its loss read like this:

ctx  = f_c(context)        # context representation
tgt  = stop_grad(f_t(target))   # target representation s — the answer key
pred = g(ctx)             # predicted representation ĝ
loss = sum((pred - tgt)**2)     # L = Σ (ĝ_i − s_i)²

Four lines, and they are the four arrows in the diagram. Line one runs the context through fcf_cfc​. Line three runs the result through ggg to get the guess. Line two runs the target through ftf_tft​ to get the answer — wrapped in stop_grad, which we are about to unpack. Line four is the loss.

The stop-gradient

stop_grad is the ╳ on the diagram. Here is what it controls.

A learning system improves by computing the loss, then nudging the boxes that produced it so the loss would have come out lower — adjusting each box’s internal numbers in the direction that helps. That backward nudge is the gradient. The stop-gradient is a wall on the target branch: gradients flow back through fcf_cfc​ and ggg and adjust them, but they hit the wall at ftf_tft​ and stop. So training updates the context encoder and the predictor, and never the target encoder directly.

Why cut off ftf_tft​ from training? Because ftf_tft​ produces the answer key, and you cannot let the system improve its score by editing the answer key. If ftf_tft​ were free to adjust, the cheapest way to drive the loss down would be to make g^\hat{g}g^​ and sss agree on anything at all — and there is a disastrously easy way to do that, which is the entire subject of M5. For now, hold one line: the stop-gradient freezes the answer key against the gradient, so the only path to a lower loss is a better guess. (How ftf_tft​ does get updated — because it is not frozen forever — is M5’s job.)

Trace one input

Numbers make this concrete. Reuse M2’s two vectors. Suppose the context flows through fcf_cfc​ and ggg and the predictor’s guess comes out as

g^=[ 0.3, 0.8, 0.2, 0.6 ],\hat{g} = [\,0.3,\ 0.8,\ 0.2,\ 0.6\,],g^​=[0.3, 0.8, 0.2, 0.6],

while the target flows through ftf_tft​ and the answer key comes out as

s=[ 0.2, 0.9, 0.1, 0.7 ].s = [\,0.2,\ 0.9,\ 0.1,\ 0.7\,].s=[0.2, 0.9, 0.1, 0.7].

Apply the loss position by position:

L=(0.3−0.2)2+(0.8−0.9)2+(0.2−0.1)2+(0.6−0.7)2=(0.1)2+(−0.1)2+(0.1)2+(−0.1)2=0.01+0.01+0.01+0.01=0.04.\begin{aligned} L &= (0.3 - 0.2)^2 + (0.8 - 0.9)^2 + (0.2 - 0.1)^2 + (0.6 - 0.7)^2 \\ &= (0.1)^2 + (-0.1)^2 + (0.1)^2 + (-0.1)^2 \\ &= 0.01 + 0.01 + 0.01 + 0.01 \\ &= 0.04. \end{aligned}L​=(0.3−0.2)2+(0.8−0.9)2+(0.2−0.1)2+(0.6−0.7)2=(0.1)2+(−0.1)2+(0.1)2+(−0.1)2=0.01+0.01+0.01+0.01=0.04.​

The loss for this input is 0.040.040.04 — small, because the guess sat close to the answer in every position. That is exactly the distance M2 computed between these same two vectors, and it is no accident: the loss is that distance, now reporting on a prediction instead of on two arbitrary inputs. Training would log this 0.040.040.04, push it lower by adjusting fcf_cfc​ and ggg, and leave ftf_tft​ alone.

Check yourself

Both encoders and the predictor produce outputs. After one training step, which boxes have changed, and which has not?

Show answer

The context encoder fcf_cfc​ and the predictor ggg change — gradients flow back through them. The target encoder ftf_tft​ does not change directly: the stop-gradient blocks the gradient on the target branch. (It does get updated, but by a separate rule covered in M5, not by this step’s gradient.)

A guess comes out as g^=[1, 0, 0, 1]\hat{g} = [1,\ 0,\ 0,\ 1]g^​=[1, 0, 0, 1] and the answer is s=[1, 1, 0, 1]s = [1,\ 1,\ 0,\ 1]s=[1, 1, 0, 1]. What is the loss?

Show answer

L=(1−1)2+(0−1)2+(0−0)2+(1−1)2=0+1+0+0=1L = (1-1)^2 + (0-1)^2 + (0-0)^2 + (1-1)^2 = 0 + 1 + 0 + 0 = 1L=(1−1)2+(0−1)2+(0−0)2+(1−1)2=0+1+0+0=1. Only the second position missed, and it missed by 1.

Exercises

1. Name the three boxes and, in one phrase each, state what each one reads and what it outputs.

Show solution
  • Context encoder fcf_cfc​ — reads the context (visible part); outputs the context representation.
  • Target encoder ftf_tft​ — reads the target (hidden part); outputs the target representation sss, the answer key.
  • Predictor ggg — reads the context representation; outputs the predicted representation g^\hat{g}g^​, the guess.

2. Compute the loss for g^=[0.5, 0.5, 0.5]\hat{g} = [0.5,\ 0.5,\ 0.5]g^​=[0.5, 0.5, 0.5] and s=[0.5, 0.4, 0.7]s = [0.5,\ 0.4,\ 0.7]s=[0.5, 0.4, 0.7].

Show solution L=(0.5−0.5)2+(0.5−0.4)2+(0.5−0.7)2=0+0.01+0.04=0.05.L = (0.5-0.5)^2 + (0.5-0.4)^2 + (0.5-0.7)^2 = 0 + 0.01 + 0.04 = 0.05.L=(0.5−0.5)2+(0.5−0.4)2+(0.5−0.7)2=0+0.01+0.04=0.05.

The first position is exact; the third miss (off by 0.20.20.2) dominates because squaring weights it 0.040.040.04 against the second’s 0.010.010.01.

3. A reader pictures the predictor’s output as a slightly blurry version of the hidden image — a small picture the system draws. Explain why that picture is wrong.

Show solution

The predictor outputs g^\hat{g}g^​, a vector — a fixed-length list of numbers, the same kind of object M2’s encoder produced. It is a point in representation space, not an image. Nothing in the architecture turns it back into pixels: there is no decoder. The system is graded by the distance between g^\hat{g}g^​ and the target representation sss, never by comparing pixels. “A blurry image” imports a decoder that JEPA does not have.

4. Suppose the stop-gradient were removed, so training could also adjust ftf_tft​. Which vector in the loss ∑i(g^i−si)2\sum_i (\hat{g}_i - s_i)^2∑i​(g^​i​−si​)2 would now be adjustable that previously was fixed?

Show solution

The target representation sss, which ftf_tft​ produces. Without the stop-gradient, training could lower the loss by moving the answer key sss toward the guess g^\hat{g}g^​, instead of only moving the guess toward the answer. That this opens a cheap, useless way to win is M5’s subject; here the point is just to see which vector the wall protects.

5. In the pseudocode, pred and tgt come from different lines. Trace each back to the input it started from.

Show solution

pred = g(ctx) and ctx = f_c(context), so pred (g^\hat{g}g^​) traces back to the context — the visible part — through fcf_cfc​ then ggg. tgt = stop_grad(f_t(target)), so tgt (sss) traces back to the target — the hidden part — through ftf_tft​ alone. Two inputs, two paths, meeting only at the loss.

Recap

A JEPA is three boxes and one number. The context encoder fcf_cfc​ turns the visible part into a representation; the predictor ggg turns that into a guess g^\hat{g}g^​; the target encoder ftf_tft​ turns the hidden part into the answer key sss. The loss L=∑i(g^i−si)2L = \sum_i (\hat{g}_i - s_i)^2L=∑i​(g^​i​−si​)2 is M2’s distance, now scoring how close the guess landed — 0.040.040.04 on our traced input. The stop-gradient freezes the answer key against training, so fcf_cfc​ and ggg learn while ftf_tft​ is left alone. That freeze is also the door to JEPA’s most dangerous failure — and to the fix that defines the architecture, which M5 takes up. First, though, M4 pays off the bet itself: why predicting a representation, rather than pixels, is the move worth making.
